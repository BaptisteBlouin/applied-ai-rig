Step 1 of 4 · Starting point

A profile is an editable shortcut, not a claim about your project. Choose one, then inspect exactly what it includes.

Step 2 of 4 · Risk assessment

Record only what you know.

Answer Yes only when the statement applies. A Yes recommends a record-keeping module; it never configures runtime protection.

How recommendations work

Your profile provides the initial set. Each Yes answer adds the related module. A No never removes a module already included by the profile, and you can toggle every optional module next.

Not a scanner: the wizard does not inspect code, discover secrets, or infer compliance.

Step 3 of 4 · Modules

Keep what earns its place.

Each module adds focused templates and registers. Selecting one creates governance records; it does not install runtime infrastructure.

The core is always present.
It adds operating principles, decisions, evidence, worklog, delivery checklist, profile, and manifest.
What does the core do?

The core separates intent, decisions, implementation history, supporting evidence, and delivery checks so humans and agents can resume work without reconstructing context.

It does not: deploy an application, monitor production, store full traces, or replace your canonical engineering tools.

What do these statuses mean?
CREATE
A new path would be added.
UPDATE
An existing managed file has a readable proposed diff.
UNCHANGED
The expected content already matches; nothing is written.
CONFLICT
Existing content cannot be safely reconciled. Refusing it cancels the whole installation.
What happens after confirmation?

The initializer prepares the complete plan, displays every affected path and readable diff, then asks for final approval. Files are staged together and rolled back if any write fails.

Confirmation here does not mean: silent overwrite, network access, deployment, publishing, or uploading project content. This mockup writes nothing.
